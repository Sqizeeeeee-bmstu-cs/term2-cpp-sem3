# Сравнение контейнеров: MyContainer vs MyHeapContainer


## Таблица сложности операций

| Операция | MyContainer (массив) | MyHeapContainer (буфер + heap) |
|----------|----------------------|--------------------------------|
| **push_back()** | O(1) амортизированная | O(1) × 5, затем O(log n) при flush |
| **pop_back()** | O(1) | O(1) |
| **operator[](index)** | O(1) | O(1) |
| **begin() / end()** | O(1) | O(1) |
| **Итератор ++/--** | O(1) | O(1) |
| **Итератор + n** | O(1) | O(1) |
| **getSize()** | O(1) | O(1) |

## Детальное сравнение операций

### push_back()

#### MyContainer (исходный)
```
Поведение:
  - Если size < capacity → добавить в конец массива (O(1))
  - Если size == capacity → выделить новый массив вдвое больший (O(n)), скопировать элементы, добавить новый

Сложность:
  - Лучший случай: O(1)
  - Худший случай: O(n) (при перераспределении)
  - Амортизированная: O(1) (за счёт экспоненциального роста capacity)
```

#### MyHeapContainer (новый)
```
Поведение:
  - Если buffer_count < 5 → добавить в buffer (O(1))
  - Если buffer_count == 5 → flushBufferToHeap + добавить в buffer
    * Перенести 5 элементов из buffer в heap: O(5) = O(1)
    * Вызвать std::make_heap: O(heap.size()) = O(n)
    * Добавить новый элемент: O(1)

Сложность:
  - Лучший случай: O(1) (добавление в не-полный буфер)
  - Худший случай: O(n) (при flush всего heap)
  - Амортизированная: O(log n) (flush происходит раз в 5 элементов)
```

**Вердикт:** MyHeapContainer лучше! 
- Избегает O(n) копирований всего массива
- Flush происходит только при переполнении буфера (раз в 5 элементов)
- Более предсказуемое поведение

---

### pop_back()

#### MyContainer
```
Просто уменьшаем size
Сложность: O(1)
```

#### MyHeapContainer
```
Просто уменьшаем buffer_count (инвариант: последний элемент всегда в буфере)
Сложность: O(1)
```

**Вывод:** Одинаково!

---

### operator[](index) — доступ по индексу

#### MyContainer
```
data[index]
Сложность: O(1) прямой доступ
```

#### MyHeapContainer
```
getElementAt(index):
  - Если index < buffer_count → возвращаем buffer[index]
  - Иначе → возвращаем heap[index - buffer_count]
Сложность: O(1) в обоих случаях
```

**Вывод:** Одинаково!

---

### Итераторы и их операции

#### MyContainer
```
- operator++/--: просто изменяем указатель (O(1))
- operator+/-: просто смещаем указатель (O(1))
- Может использовать random_access_iterator_tag
```

#### MyHeapContainer
```
- operator++/--: просто изменяем current_pos (O(1))
- operator+/-: просто смещаем current_pos (O(1))
- Доступ к элементу через getElementAt (O(1))
- Тоже random_access_iterator_tag
```

**Вывод:** Идентичны

## Сравнение по памяти

| Аспект | MyContainer | MyHeapContainer |
|--------|-------------|-----------------|
| **Памяти для n элементов** | n × sizeof(T) | n × sizeof(T) |
| **Служебная память** | sizeof(int) × 2 (size, capacity) | sizeof(int) × 2 + sizeof(vector) + 5 × sizeof(T) |
| **Фрагментация памяти** | Одно выделение | Два выделения (buffer + heap) |
| **Cache-friendly** | Лучше (один непрерывный блок в памяти) | Хуже (два отдельных блока) |

**Вывод:** MyContainer лучше по памяти и cache-локальности. MyHeapContainer платит небольшой overhead за оптимизацию вставок.

---

## Общее сравнение

### Преимущества MyHeapContainer
**push_back() избегает O(n) операций**
  - Вместо копирования всего массива при переполнении
  - Flush только раз в 5 элементов
  - Более предсказуемое время отклика

**Лучше для workload-ов с частыми вставками**
  - Нет резких скачков в时истории вставок
  - Амортизированное логарифмическое время



### Недостатки MyHeapContainer
**Больше служебной памяти**
  - Дополнительный buffer + vector overhead

**Хуже cache-локальность**
  - Элементы живут в двух местах памяти

**Сложнее в отладке**
  - Нужно учитывать инварианты buffer/heap

**Heap не упорядочена**
  - Если нужен упорядоченный доступ, нужна доп логика

---

## Практические рекомендации

| Сценарий | Рекомендация |
|----------|--------------|
| Много вставок подряд | MyHeapContainer |
| Случайные операции | MyContainer (проще, меньше памяти) |
| Требуется max/min быстро | MyHeapContainer (но это не реализовано) |
| Нужна упорядоченность | Оба контейнера (нужно std::set/map) |
| Очень частые вставки/удаления | MyHeapContainer (но нужна оптимизация) |
| Всегда нужен прямой доступ | Оба работают O(1) |

---

## Выводы

**MyHeapContainer успешно решает проблему O(n) реаллокации** при push_back за счёт буфера.

Цена оптимизации:
- +свежий buffer на стеке (5 элементов)
- +vector overhead для heap
- -немного хуже с cache-локальностью

Но выигрыш в **предсказуемости и скорости вставок** делает это компромиссом стоящим, особенно для контейнеров, которые растут динамически.
